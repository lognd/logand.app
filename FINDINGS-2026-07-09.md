## LOW

### L1 -- Uncaught Stripe SDK errors on the /pay calls surface as a raw 500 [FIXED]

- **Where**: `backend/src/logand_backend/api/invoices_public.py` -- `pay_invoice`
  (`PaymentIntent.retrieve` / `.cancel` / `.create`) and `pay_invoice_via_paypal`
  (`.retrieve`).
- **What's wrong**: These `asyncio.to_thread(stripe.*)` calls are unwrapped.
  `domain/invoices/refunds.py::refund_payment` and
  `api/webhooks.py::_charge_billing_postal_code` both wrap Stripe calls in
  `try/except stripe.error.StripeError`; these customer-facing pay entry points
  do not, so a transient `APIConnectionError`/`RateLimitError` propagates as an
  unhandled 500.
- **Failure scenario**: Customer clicks "Pay with card"; Stripe blips during
  `PaymentIntent.create`; FastAPI returns a bare 500 instead of the
  503/"try again shortly" path the code takes care to present elsewhere. No
  money is lost or double-moved.
- **Fix direction**: Wrap the three `pay_invoice` Stripe calls and the
  `pay_invoice_via_paypal` retrieve in `try/except stripe.error.StripeError`,
  raising `HTTPException(503, ...)` matching the existing convention.

### L2 -- PENDING PayPal capture with a mismatched amount is recorded without a needs-review flag [ACCEPTED, not fixed]

- **Where**: `backend/src/logand_backend/api/invoices_public.py`
  `capture_invoice_paypal_payment`, PENDING branch.
- **What's wrong**: The COMPLETED branch computes `overpaid` and calls
  `flag_invoice_needs_review` on an amount mismatch; the PENDING branch records
  the pending Payment with no equivalent check.
- **Failure scenario**: None constructible. `service.py::reconcile_pending_paypal_captures`'s
  `paid_so_far > amount_total` check fires on reconciliation, so the condition is
  caught, just later. This is branch-symmetry / earlier-visibility, not
  demonstrated money loss.
- **Decision**: Left as-is. Adding a second overpayment code path here would
  duplicate the reconciler's rule (see CLAUDE.md "NO DUPLICATION" -- two copies
  of a rule is a bug waiting to desync) to buy only earlier visibility of a
  condition already reliably caught. Recorded rather than silently dropped.

## Found during verification, not by the audit

### M1 -- The test suite silently loads a developer's real `backend/.env`, so local runs disagree with CI [FIXED]

- **Where**: `backend/src/logand_backend/app/config.py::AppConfig.from_external`
  calls `load_dotenv()` on EVERY invocation (including inside request
  handlers), and `backend/tests/conftest.py` did nothing to stop it.
- **What's wrong**: `load_dotenv` does not overwrite a variable already in
  `os.environ`, which makes it look harmless. But a test that does
  `monkeypatch.delenv("STRIPE_PUBLISHABLE_KEY")` to exercise an
  "unconfigured" code path removes the variable -- so the next
  `from_external()` inside the route re-reads `.env` and puts it straight
  back, undoing the test's own setup mid-request.
- **Failure scenario**: With a populated `backend/.env`, running
  `pytest tests/system/test_stripe_config.py tests/system/test_zelle_config.py
  tests/system/test_admin_logs_api.py tests/system/test_request_logging.py`
  fails 8 tests locally while passing in CI (which has no `.env`). Confirmed
  by stashing all work and re-running against a clean `HEAD` in the same
  working tree: the same 8 fail. A test suite whose result depends on an
  untracked, gitignored file is worse than one that simply fails -- it
  trains you to ignore red.
- **Fix applied**: New autouse fixture `_no_dotenv_reinjection` in
  `backend/tests/conftest.py` monkeypatches `config.load_dotenv` to a no-op
  for the duration of each test. Variables already loaded at import time
  (notably `DATABASE_URL`, which the db fixtures genuinely source from
  `.env`) stay in `os.environ`; only the mid-test re-injection is stopped.
  All 8 now pass, and the full suite is 546 passed / 3 skipped / 0 failed.

### M2 -- Migration 0028's downgrade could not run once a contact row existed [FIXED]

- **Where**: `backend/src/logand_backend/db/migrations/versions/0028_contact_users_verification.py::downgrade`.
- **What's wrong**: `ALTER COLUMN password_hash SET NOT NULL` cannot execute
  while any contact row (`password_hash IS NULL`) exists. Deleting those rows
  is not an option either -- `invoices.customer_id` is `ON DELETE RESTRICT`, so
  an invoiced contact blocks the rollback entirely.
- **Failure scenario**: Deploy 0028, invoice one stranger by email (creating a
  contact row), then hit any unrelated problem and try to roll back. `alembic
  downgrade` dies on `null value in column "password_hash"` -- precisely when
  an operator most needs the rollback to work.
- **How it was found**: The test suite builds its schema with
  `Base.metadata.create_all()` and never executes the migration chain, so no
  test covered this. Caught by running the real chain against a throwaway
  Postgres container.
- **Fix applied**: `downgrade` now assigns each contact row a valid argon2 hash
  of a fresh random password nobody holds (not a sentinel like `'!'` -- the
  pre-0028 login path calls `verify_password` unconditionally, and a malformed
  hash raises rather than cleanly returning False). Rows survive, keep their
  invoices, and nobody can authenticate as them.

### Note -- the migration chain forked at 0021

The remote's tax work and this work both added a migration with
`down_revision = "0021_payment_zip_code"`, giving Alembic two heads and making
`alembic upgrade head` fail on deploy. Resolved by renumbering the
contact-users migration to `0028`, chained after `0027_tax_rule_citation`.
Verified end to end against a real Postgres: upgrade `0027 -> 0028` backfills a
legacy user so it can still log in, contact rows are insertable,
`ck_users_contact_or_active` rejects a verified passwordless row, and the
downgrade round-trips.

## Verified correct (do not re-audit)

- **Migration chain**: 0000<-...<-0021<-0022 intact. 0020's `revision` is the
  truncated `"0020_password_reset_tok"` and 0021's `down_revision` points at
  exactly that; single head, no forks, no duplicate ids.
- **Stripe webhook**: signature-verified, idempotent on
  `Payment.stripe_payment_intent_id` + the 0003 partial unique index
  (SAVEPOINT/IntegrityError backstop); out-of-order `payment_failed`-after-
  `succeeded` no-ops; locks released before email fan-out.
- **Refunds**: `client_request_id` doubles as Refund id and provider idempotency
  key; retry short-circuits sound.
- **Settlement math**: `get_paid_so_far` nets partially-refunded remainders;
  `has_pending_payment` guards every initiation path.
- **Stats**: `net_collected` avoids double-subtracting refunded + lost-dispute
  payments; all money queries filter `deleted_at.is_(None)`.
- **Auth infra**: `sessions.py` enforces expiry AND `disabled_at`, sliding idle
  capped by absolute lifetime, sha256-only token storage; `csrf.py` binds
  double-submit to session `csrf_secret` with constant-time compares;
  `rate_limit.py` uses rightmost XFF and fails open on a Redis outage.
- **IDOR**: customer routes go through `_get_owned_invoice` (404 not 403,
  server-derived customer_id); admin routes uniformly `require_admin`;
  `admin_data` uses `Base.metadata.tables` lookups with parameterized column
  comparisons.
- **Uploads/downloads**: `safe_filename`, `_safe_log_path`, `read_upload_capped`
  all sound; email templates HTML-escape admin free text; `Pay.tsx` uses React
  text nodes (no `dangerouslySetInnerHTML`).
- **Ops/CI/deploy**: compose reads all secrets from `backend/.env` via
  `env_file`; Caddyfile caps `/api/*` at 25MB; `deploy.yml` polls all four CI
  check-runs to terminal state; no secret echoed.
