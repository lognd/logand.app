# 03 -- Database

Audience: anyone writing SQLAlchemy models, Alembic migrations, or
queries. Read [00-overview.md](00-overview.md) first. For role/access
rules referenced below, see [02-auth-and-security.md](02-auth-and-security.md).

## Engine

PostgreSQL (locked decision, see [00-overview.md](00-overview.md)).
SQLAlchemy 2.x async ORM, `asyncpg` driver. Alembic for migrations,
autogenerate-then-review (never hand-skip review of an autogenerated
migration).

## Conventions

- Every table: `id` (UUID v4 primary key, not serial -- avoids leaking
  row counts/sequence to anyone who sees an ID), `created_at`,
  `updated_at` (both `timestamptz`, server-default `now()`,
  `updated_at` maintained by an `onupdate` trigger or SQLAlchemy hook).
- Money: `numeric(12, 2)` columns, never `float`. Currency is USD-only
  at v1 -- no currency column needed yet, but leave a comment marking
  where it'd go if this ever changes.
- Soft-delete only where audit history matters (invoices, budget
  entries) via `deleted_at timestamptz null`; hard-delete elsewhere
  (sessions, inventory items) -- audit trail has a real cost, don't pay
  it where it isn't needed.
- Foreign keys always `ON DELETE RESTRICT` unless a doc below explicitly
  says cascade -- silent cascading deletes are how audit trails die.

## Core tables

### `users`

One admin row ever exists (enforced at the app layer, not a DB
constraint -- a unique partial index on `role = 'admin'` is acceptable
if you want a belt-and-suspenders check).

```
id                uuid pk
email             text unique not null
password_hash     text not null        -- Argon2id, see 02
role              text not null check (role in ('admin', 'customer'))
created_at        timestamptz
updated_at        timestamptz
```

### `sessions`

```
id                uuid pk
user_id           uuid fk -> users.id, on delete cascade
token_hash        text unique not null   -- sha256 of the raw session token
csrf_secret       text not null
expires_at        timestamptz not null
created_at        timestamptz
last_seen_at      timestamptz            -- updated on each request, drives sliding expiry
```

Sessions cascade-delete with their user -- there's no audit value in
keeping a dead user's sessions around.

## Migrations

- One Alembic revision per logical change -- don't bundle an invoices
  migration with an unrelated inventory migration.
- Every migration must have a working `downgrade()`. If a downgrade is
  genuinely destructive (e.g. dropping a column with data), raise
  `NotImplementedError` in `downgrade()` with a comment explaining why,
  rather than silently no-op-ing.
- Run via `make migrate` (backend Makefile, see
  [01-backend-architecture.md](01-backend-architecture.md)).
